# SHIFT Platform Review & Viability Study — July 2026

**Scope:** Full review of the codebase as of 2026-07-19 (last modified Nov 2025 – Jan 2026). Both modules compile cleanly on Go 1.25. Zero test files exist. The project has never been committed to git.

> **Outcome (2026-07-19):** The clean-repository recommendation was accepted. The prototype now lives in `_archive/` (paths below are relative to it); compiled binaries and `.so` files were deleted; the schema was preserved at `docs/reference/schema-v0.sql`.

**Verdict up front:** The repository is a Phase 1–2 prototype (~5.5k LOC) whose most valuable assets are the *design documents* and the *PostgreSQL schema*, not the code. The three load-bearing architectural choices actually on disk — Go `.so` plugin connectors, per-runner private SQLite "distributed" queues, and a fully-buffered `map[string]interface{}` execution engine — each need replacement, and the third is the heart of the product. **Recommendation: start a clean repository**, carrying forward the design docs, schema, and a small set of proven patterns. Details in §5.

## 2. What's genuinely good (worth carrying forward)

1. **The design documents** (`plan.md`, `agents.md`, `security_spec.md`, `api_contracts.md`, `data_models.md`) — coherent hub-and-spoke vision, sensible phasing, correct security posture *on paper*.
2. **The PostgreSQL schema** (`hub/migrations/schema.sql`) — a forward-looking target data model: accounts → users → roles/RBAC, runner groups with versioned configs + hashed secrets, versioned flows and connectors, billing profiles, execution telemetry with resource columns.
3. **Control-plane/data-plane split** — hub persists/pushes config and receives telemetry; runners execute. This is the right shape and matches how Workato/Boomi agents work.
4. **Atomic task claim** — `UPDATE tasks … WHERE status='pending' … RETURNING` (`taskqueue/queue.go:111`) is the correct claim idiom.
5. **Versioned connector distribution with checksums** — right instinct (needs signing, not just hashing).
6. **Parameterized SQL throughout** — no injection anywhere.
7. **Small dependency footprint** — stdlib `net/http`, gorilla/websocket, pgx, sqlite3, cron. No framework sprawl.

## 3. Critical defects

### 3.1 The execution engine contradicts the core product goal
The stated differentiator is *memory-efficient, disk-light transformation*. The implementation is the opposite:

- HTTP connector `io.ReadAll`s entire response bodies (`connector/http.go:137`).
- Every step fully unmarshals its input into `map[string]interface{}` and re-marshals output — reflection-heavy JSON on the hot path.
- The result envelope re-embeds the full input + every intermediate step output + final output, then marshals it all again (`execution/engine.go:243-264`) — payloads duplicated several times in memory per execution.
- Every task's input *and* output payload is written to SQLite as TEXT (`taskqueue/queue.go`) — the disk-heavy pattern the project exists to avoid.
- No `io.Reader` pipelines, no streaming primitives, anywhere.
- Task goroutines are spawned with no concurrency cap (`main.go:567`).
- Reported metrics are fabricated: "CPU time" is wall-clock; "memory used" is a racy delta of process-global `runtime.MemStats.Alloc` (`engine.go:141-156`).

This is not a tuning problem — the engine's data model *is* the buffering. Fixing it means rewriting the engine's core.

### 3.2 "Distributed" task queue isn't distributed
Each runner's SQLite DB lives at `os.TempDir()/shift-runner-<id>.db` (`main.go:48`) — private per runner. The scheduler's coordination design (deterministic task IDs + duplicate-insert dedup + load-proportional delay) assumes a *shared* store, so **every runner in a group independently executes every scheduled flow**. The only real sharding is the Kafka consumer group — which was never in the design docs, runs PLAINTEXT, and pulls in a heavyweight infra dependency the plan explicitly avoided.

### 3.3 Go `plugin` connectors are a dead end
`-buildmode=plugin` `.so` files require the **exact same Go toolchain version, dependency graph, and build flags** as the host binary (the build script itself warns about this), are Linux/macOS-only, can never be unloaded (memory grows per version), and force CGO. The download URL encodes GOOS but not Go version — any toolchain skew fails at `plugin.Open`. This blocks the entire connector-ecosystem strategy: third parties can't realistically ship compatible binaries.

### 3.4 Security is absent (not weak — absent)
- Zero authentication on any hub or runner endpoint; WS `CheckOrigin` returns `true` everywhere.
- Runner identity spoofable via query params (`runner-service/main.go:153-183`); the schema's `api_key_hash`/`registration_token` columns are never validated.
- Group secret generated and hashed correctly, then **stored and transmitted in plaintext** inside the config JSONB and pushed over the unauthenticated WS (`runnergroup/manager.go:117-119`) — defeating its own design.
- Unauthenticated `.so` download over plain HTTP, checksum delivered on the same channel → **remote code execution by design** for anyone who can MITM or reach the hub.
- Step auth credentials (passwords, bearer tokens, API keys) flow through step config, get written to world-readable SQLite in `/tmp`, and are echoed back to the hub in result envelopes.
- HTTP connector has no SSRF protection.

### 3.5 Correctness bugs
- **Data races:** connector `Registry` map has no mutex (`connector/interface.go:20-40`) with cross-goroutine Register/Get; both WS hubs mutate client maps and close channels under `RLock` (`hub/websocket/hub.go:87-96`, `runner/websocket/p2p.go:72-82`) — double-close panics possible.
- **Hub client never reconnects** after an established connection drops (`main.go:495-505` retries only until first success) — one blip permanently severs a runner.
- Full send-buffers silently drop messages, including execution status reports (`hub_client.go:60-66`).
- P2P re-broadcasts every message to all peers with no dedup/TTL — message amplification if the mesh ever forms.
- Duplicate-key detection by string-matching error text (`queue.go:91-92`).
- Errors swallowed with `_` throughout; goroutine leak in stuck-task recovery (`main.go:526-555`).

## 5. Viability study: evolve in place vs clean repository

### The case for evolving in place
- Both modules compile; the repo runs end-to-end (schedule → Kafka → execute → report) as a demo.
- Managers/persistence layers are cleanly factored and reusable.
- The schema is good and already written.

### The case for a clean repository
1. **The heart must be rewritten anyway.** The execution engine's buffered-map data model cannot be retrofitted into a streaming engine — the engine, connector interface (`Execute(ctx, action, inputJSON []byte) ([]byte, error)` is inherently buffer-in/buffer-out), and queue payload model all change together.
2. **Two of three structural choices are being discarded** (Go plugin `.so`, per-runner SQLite masquerading as distributed; Kafka is also questionable for an "easy to self-host" spoke). What remains is scaffolding.
3. **Nothing is load-bearing.** No git history to preserve, no users, no tests encoding behavior. The usual cost of a rewrite — losing accumulated correctness — doesn't apply, because there is no accumulated correctness.
4. **Security cannot be bolted on.** Identity, secrets handling, and transport auth need to be present from the first commit of a platform whose pitch includes "Encrypted by Default."
5. **~5.5k LOC is a weekend, not an asset.** Reproducing the salvageable parts (schema, claim idiom, WS message shapes) is far cheaper than untangling the simulation code, dead code (~10 identified dead functions/types), and races.

### Recommendation
**Clean repository.** Treat SHIFT-as-it-stands as a successful *learning prototype*: it validated the hub-and-spoke shape and surfaced exactly which mechanisms don't work. Carry forward:

- All five design docs (revised: drop the Kafka accident, decide the connector model deliberately).
- `schema.sql` as the target hub data model (add audit + secrets tables).
- Patterns: atomic claim via `UPDATE…RETURNING`, hub-pushes/runner-reports message protocol, versioned+checksummed connector distribution (add signing), docker-compose dev topology.
- This review as the "lessons learned" register.

### Key decisions the clean build must make first (in dependency order)
1. **Streaming data plane** — the differentiator. Design the inter-step data contract as `io.Reader`-based streams with typed record batches, spill-to-disk only above a memory watermark. Everything else hangs off this contract. *Get this right before writing any connector.*
2. **Connector isolation model** — replace Go `plugin`. Two credible options: **hashicorp/go-plugin (gRPC subprocess)** — process isolation, crash containment, any-language connectors, streaming via gRPC streams; or **WASM (wazero)** — pure-Go, no CGO, sandboxed, per-tenant safe, very AI-toolchain-friendly, but streaming ergonomics are younger. A hybrid (WASM for transforms, subprocess for connectors) is also defensible.
3. **Task distribution** — pick one deliberate mechanism: a genuinely shared queue (Postgres `SKIP LOCKED` per group, or embedded NATS JetStream in runners) rather than Kafka-for-a-demo. Must honor the "lightweight self-hosted spoke" constraint.
4. **Identity from commit one** — OIDC on the hub, token-based runner registration, mTLS or signed-token WS auth, secrets encrypted at rest (age/AES-GCM envelope) and never echoed in payloads/results.
5. **Tests and CI from commit one** — the prototype's zero-test state is how simulation code and races survived unnoticed.

### What "performance exceeds the incumbents" actually requires
The Go-beats-JVM/proprietary-runtime thesis is sound *only if* the engine is: (a) streaming end-to-end (no whole-payload buffering), (b) zero-copy where possible with pooled buffers, (c) disk-touching only on explicit spill or durability boundaries, (d) bounded concurrency with backpressure, and (e) measured honestly (per-step CPU/alloc profiling, not the current fabricated metrics). Build benchmark harnesses (large-payload transform throughput, records/sec/GB RAM) alongside the engine from the start — that's the marketing artifact as much as the engineering guardrail.
